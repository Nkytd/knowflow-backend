# 企业级能力升级说明

这份文档记录 KnowFlow 从“功能可演示”继续往“更像生产系统”推进的三类能力：SLA 工单治理、AI 调用成本观测、Agent 工具调用。它们都是面试 Java 后端开发时比较好讲的工程化亮点。

## 1. SLA 工单治理

### 业务目标

工单系统不能只有“创建、处理、关闭”，还需要回答：

- 高优先级问题有没有在承诺时间内处理？
- 哪些工单已经超时？
- 哪些工单临近超时，需要优先跟进？
- 超时记录能否沉淀到流转轨迹，便于复盘？

### 已实现能力

- 工单创建时根据优先级自动生成 SLA 策略和截止时间。
- 工单列表支持按 `slaStatus` 筛选。
- 工单列表和详情展示 SLA 策略、状态、截止时间、剩余时间。
- 定时任务扫描未完结工单，超时后标记为 `BREACHED`。
- 超时动作写入 `ticket_flow`，动作类型为 `SLA_BREACHED`。

### 策略映射

| 优先级 | SLA 策略 | 承诺处理时间 |
|---|---|---|
| `URGENT` | `P1-4H` | 4 小时 |
| `HIGH` | `P2-8H` | 8 小时 |
| `MEDIUM` | `P3-24H` | 24 小时 |
| `LOW` | `P4-72H` | 72 小时 |

### 面试讲法

可以这样讲：

> 我没有把工单只做成 CRUD，而是补了 SLA 治理。工单创建时会根据优先级生成 SLA 截止时间，后台有定时扫描任务判断是否超时，超时后更新工单 SLA 状态并写入流转轨迹。管理台可以按 SLA 状态筛选，也能在详情页看到策略、截止时间和剩余时间。这个设计让工单系统具备了真实客服/ITSM 系统里的时效管理能力。

## 2. AI Token 与成本观测

### 业务目标

AI 应用上线后，除了回答质量，还要关注成本和稳定性：

- 最近调用了多少次模型？
- 成功率、失败率、未命中率是多少？
- 输入/输出 Token 消耗多少？
- 不同模型的平均延迟和估算成本是多少？
- 是否存在模型调用失败率升高的问题？

### 已实现能力

- 新增管理端接口：`GET /api/v1/admin/ops/ai/usage?days=7`
- 统计最近 N 天问答/AI 调用次数、成功数、失败数、未命中数、转人工数。
- 统计输入 Token、输出 Token、总 Token。
- 统计平均延迟、平均检索耗时、平均生成耗时。
- 按模型聚合调用量、成功率、Token、平均延迟、估算成本。
- 运维健康页展示 AI 调用治理卡片和模型明细表。

### 成本估算配置

```yaml
knowflow:
  ops:
    ai-cost:
      input-cny-per-1k: 0.002
      output-cny-per-1k: 0.006
```

### 面试讲法

可以这样讲：

> RAG 系统不是只要能调用大模型就结束了，企业会关心 Token 成本和模型稳定性。我在问答记录上沉淀了模型名、输入输出 Token、检索耗时、生成耗时和回答状态，然后做了一个 AI usage 统计接口，管理台可以看到近 7 天调用量、成功率、未命中率、转人工率、Token 消耗和模型维度成本估算。这个能力可以继续接入 Prometheus 或企业成本看板。

## 3. Agent 工具调用

### 业务目标

智能助手不能只聊天，还应该能执行受控业务动作。例如：

- 查询工单进度
- 创建工单
- 查看我的未完结工单
- 查询可用知识库
- 从已解决工单生成知识草稿

### 已实现接口

```http
GET /api/v1/app/agent/tools
POST /api/v1/app/agent/tools/execute
```

### 已实现工具

| 工具编码 | 能力 |
|---|---|
| `QUERY_TICKET_PROGRESS` | 根据 `ticketId` 或 `ticketNo` 查询工单进度 |
| `CREATE_TICKET_FROM_QA` | 从问答记录创建人工工单 |
| `LIST_MY_OPEN_TICKETS` | 查询当前用户未完结工单 |
| `LIST_KNOWLEDGE_BASES` | 查询当前租户启用中的知识库 |
| `GENERATE_KNOWLEDGE_DRAFT_FROM_TICKET` | 从已解决工单生成知识草稿 |

### 当前边界

现在实现的是“工具注册表 + 受控执行器”，还不是完整自主 Agent Planner。也就是说，系统已经具备让大模型调用业务工具的后端基础，但还没有把工具选择、参数生成、结果反思全部交给大模型自动编排。

这个边界在面试里要主动讲清楚，显得更专业：

> 当前我先实现了 Agent Tool Registry 和 Tool Executor，保证工具权限、参数校验和业务执行链路是可靠的。下一步可以接 OpenAI-compatible function calling，让模型根据用户意图选择工具，再由后端执行工具并把结果回填给模型生成最终回复。

## 4. 本次验证点

新增集成测试覆盖了 Agent 工具链路：

- 登录租户管理员。
- 创建测试知识库。
- 查询 Agent 工具列表。
- 执行 `LIST_KNOWLEDGE_BASES`，验证可用知识库返回。
- 创建问答会话和问答消息。
- 执行 `CREATE_TICKET_FROM_QA`，验证工单创建与 SLA 初始化。
- 执行 `QUERY_TICKET_PROGRESS`，验证可查询到工单和 SLA 截止信息。
