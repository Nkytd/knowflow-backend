# KnowFlow 最终演示脚本

> 目标：用 8 到 12 分钟讲清楚 KnowFlow 不是单纯聊天页，而是“知识库建设 -> RAG 问答 -> 工单兜底 -> 知识回流 -> 运营治理”的企业级闭环。

## 1. 演示前准备

推荐运行方式：

```text
IDEA Run Configuration: KnowFlow - Dev (8081 Fallback)
Profile: dev
Port: 8081
```

依赖容器：

```text
knowflow-mysql
knowflow-redis
knowflow-rabbitmq
knowflow-minio
```

演示入口：

```text
http://127.0.0.1:8081/admin/dashboard
```

常用账号：

```text
tenant.admin / Tenant@123
knowledge.operator / Tenant@123
```

演示前检查：

- `.run` 中不要保留真实大模型 API Key。
- Docker 日常停止使用 `docker compose stop`，不要用 `docker compose down -v`。
- 浏览器如果样式没有刷新，先强制刷新页面。
- 优先使用 `8081`，避免和旧的 `8080` Java 进程混用。

## 2. 开场定位

推荐开场话术：

```text
KnowFlow 是一个面向企业内部知识服务的 Java 后端项目。
它不是只做一个聊天机器人，而是把文档入库、解析切片、RAG 问答、未命中转工单、工单解决后知识回流和运营治理串成闭环。
我会按一条真实业务链路演示：知识进入系统、用户提问、AI 无法解决时转人工、人工结果再沉淀回知识库。
```

重点强调：

- 业务价值：减少重复咨询，让知识持续沉淀。
- 后端价值：多租户、RBAC、异步任务、失败治理、审计、可观测性。
- AI 价值：RAG 不是只调模型，而是检索、上下文、置信度和来源追踪的组合。

## 3. 演示主线

### 第一步：运营看板总览

页面：

```text
/admin/dashboard
```

操作：

- 登录 `tenant.admin / Tenant@123`。
- 展示顶部全局导航：运营看板、智能问答入口、知识运营、工单协同、系统治理。
- 展示问答趋势、热点问题、未命中、工单与知识草稿转化。

讲解点：

```text
这个页面不是为了炫图表，而是作为业务闭环的入口：我可以看到哪些问题被知识库解决，哪些问题还需要人工，哪些人工结果已经转成知识草稿。
```

### 第二步：知识库和文档入库

页面：

```text
/admin/knowledge-bases
/admin/documents
```

操作：

- 进入“知识运营”。
- 查看知识库状态、文档统计、解析失败统计。
- 进入“文档管理”，展示上传、预览、下载、批量操作入口。

讲解点：

```text
企业知识先以文档形式进入系统。后端会把元数据落到 MySQL，把文件交给 MinIO，并创建解析任务，而不是在上传接口里同步阻塞解析。
```

### 第三步：解析任务与异步治理

页面：

```text
/admin/parse-tasks
/admin/dead-letters
```

操作：

- 在“解析任务”查看任务状态。
- 展示 `PENDING -> PROCESSING -> SUCCESS/FAILED` 状态推进。
- 如果存在失败任务，跳转“死信治理”查看失败原因、重试和回放入口。

讲解点：

```text
解析和索引是长耗时任务，所以用 RabbitMQ 解耦。任务状态会持久化到 parse_task，失败消息进入 dead_letter_message，运维页面可以治理，而不是失败后悄悄丢掉。
```

### 第四步：智能问答与来源追踪

页面：

```text
/assistant
```

操作：

- 选择知识库和模型。
- 输入一个知识库可以命中的问题。
- 展示回答、引用来源、检索调试信息。
- 展开 query variants、lexical/vector/final score。

讲解点：

```text
RAG 的重点不是把问题直接扔给大模型，而是先做查询扩展和混合检索，召回 chunk 后构造上下文，再让模型回答。页面上的来源和分数用于解释答案为什么可信。
```

### 第五步：未命中转工单

页面：

```text
/assistant
/admin/qa-records
/admin/tickets
```

操作：

- 输入一个知识库无法覆盖的问题，或手动点击转人工。
- 创建工单。
- 从问答入口或问答记录跳转到关联工单。
- 展示工单状态、处理记录、SLA 信息。

讲解点：

```text
AI 系统不能假装什么都能回答。低置信度或未命中时进入人工工单，既能兜底用户问题，也能反向暴露知识库缺口。
```

### 第六步：工单解决后知识回流

页面：

```text
/admin/tickets
/admin/knowledge-drafts
```

操作：

- 在工单详情里查看处理记录。
- 将工单标记为已解决。
- 查看生成的知识草稿。
- 展示草稿审核、发布入口。

讲解点：

```text
工单不是闭环终点。人工解决方案会生成知识草稿，运营审核后再发布回知识库。后续类似问题就能被 RAG 命中，形成持续优化。
```

### 第七步：运维健康与审计

页面：

```text
/admin/ops-health
/admin/audit-logs
```

操作：

- 展示任务成功率、失败率、P95 耗时、DLQ 数量。
- 展示 AI 用量、Token 和成本估算。
- 打开审计日志查看关键操作记录。

讲解点：

```text
企业系统不只要能跑，还要能治理。这里把异步任务、依赖健康、AI 调用和审计都放进管理台，方便定位问题和解释系统运行状态。
```

## 4. 技术讲解顺序

推荐顺序：

1. 业务闭环：文档入库 -> RAG 问答 -> 转工单 -> 知识回流。
2. 后端分层：Controller -> Service -> Mapper -> MySQL/H2 + Flyway。
3. 鉴权与租户：JWT、RBAC、`tenant_id` 数据隔离。
4. 异步链路：RabbitMQ 解析任务、Redis 运行态辅助、DLQ 治理。
5. RAG 链路：chunk、混合检索、上下文构造、来源追踪、低置信度兜底。
6. 工程化：Docker Compose、Smoke Test、Actuator、GitHub Actions。

## 5. 常见追问

### 为什么不用单纯向量数据库？

```text
当前阶段先做轻量级混合检索，结合 lexical score、vector score、标题章节加权和单文档数量控制，能把 RAG 关键链路讲清楚。
后续可以接入专业向量数据库和 rerank 模型，但不会改变现有业务闭环。
```

### 为什么要有工单？

```text
企业知识库一定会有覆盖不足的时候。如果没有人工兜底，AI 就容易胡编或者让用户无处可去。
工单能解决未命中问题，同时把这些问题沉淀成知识库优化素材。
```

### 怎么体现 Java 后端能力？

```text
项目里有 RBAC、JWT、多租户、MyBatis-Plus、Flyway、异步任务、DLQ、MinIO、Redis、RabbitMQ、审计日志、Actuator 和 Smoke Test。
这些能力说明它不是单页 demo，而是按企业后台系统的方式设计。
```

### 怎么保证失败可治理？

```text
解析任务状态会持久化，失败消息进入 DLQ，页面能查看错误原因、关联文档和任务，并支持重试或回放。
运维健康页还能看到失败率、P95 耗时和依赖健康。
```

## 6. 收尾话术

```text
这个项目最核心的设计不是让大模型回答一次问题，而是让企业知识持续流动起来。
文档进入系统后能被问答使用；问答解决不了会进入人工；人工解决后又沉淀回知识库；运营和运维页面再持续观察这个闭环是否健康。
这也是我希望展示的后端能力：不仅能写接口，还能把业务、AI 链路和工程治理组织成一个可维护的系统。
```
