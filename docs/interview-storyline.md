# 面试讲述话术

## 1 分钟版本

我做的是一个企业级智能知识服务与工单协同平台。它的核心目标是让企业内部常见问题先走知识库问答，如果检索不到可靠答案，就自动转人工工单；工单解决后，再沉淀成知识草稿回流知识库，形成闭环。技术上我主要用 Spring Boot、MyBatis-Plus、JWT、RabbitMQ、Redis、MinIO 和 OpenAI Compatible 模型接口，实现了多租户、文档解析、混合检索、工单流转和运营分析。

## 3 分钟版本

这个项目一开始并不是为了做一个简单问答页面，而是为了做一个更完整的知识服务平台。业务上，它把知识库、问答、工单、知识回流、运营分析连接起来。技术上，我把系统拆成几条主链路：

1. 鉴权和租户管理链路，保证不同租户的数据隔离。
2. 文档接入链路，用户上传文档后进入解析任务，异步切 chunk，再做向量索引。
3. 问答链路，用户提问后先做词法+向量混合检索，再把命中的上下文送给大模型生成答案。
4. 治理链路，如果低置信度或未命中，就转人工工单；工单解决后可以反向生成知识草稿，审核发布后回流知识库。
5. 运营链路，后台看板能看未命中问题、工单转化、失败任务和知识回流情况。

我觉得这个项目最能体现能力的地方，是它不是单点功能，而是完整业务闭环，而且把 AI 应用里最关键的检索质量、任务稳定性和工程化交付都覆盖到了。

## 2 到 3 分钟面试压缩版

我这个项目叫 KnowFlow，定位是企业级智能知识服务与工单协同平台。它不是单纯做一个聊天页，而是把企业内部知识从“进入系统”到“持续沉淀”串成闭环：管理员先创建知识库并上传文档，系统把文件交给 MinIO，元数据落 MySQL，然后通过 RabbitMQ 创建异步解析任务，把文档切成 chunk 并建立检索索引。用户提问时，后端先做查询扩展和混合检索，再把命中的知识片段作为上下文交给大模型生成回答，同时保留来源、召回分数和问答记录。

如果知识库没有可靠命中，我没有让 AI 硬答，而是设计了转人工工单。工单处理完以后，可以生成知识草稿，运营审核发布后再回流知识库。这样系统就形成了“文档入库 -> RAG 问答 -> 未命中转工单 -> 工单解决 -> 知识回流 -> 运营观测”的业务闭环。

工程上我重点做了几块：第一是安全边界，用 Spring Security、JWT、RBAC 和 `tenant_id` 做多租户隔离，并写了集成测试覆盖匿名访问、越权访问和跨租户数据隔离；第二是异步可靠性，解析任务用 RabbitMQ 解耦，用 Redis 和数据库状态控制运行态，失败进入 DLQ，支持重试和回放，同时用状态机和 attempt 栅栏避免重复消息或旧 worker 回写污染任务；第三是质量观测，后台有运营看板、未命中分析、问答记录、运维健康、AI Token 和成本统计、审计日志，也有 smoke test 和检索评估文档支撑演示。

所以我介绍这个项目时会强调：它展示的不只是“会调用大模型”，而是我能把 Java 后端的鉴权、数据建模、异步任务、失败治理、测试验证和 AI RAG 链路组织成一个可演示、可追踪、可继续演进的企业系统。

### 30 秒追问版

如果面试官追问亮点，我会压缩成三句话：第一，业务闭环完整，AI 回答不了会转工单，工单解决后还能反哺知识库；第二，工程链路完整，MySQL、MinIO、RabbitMQ、Redis、Flyway、JWT 和 Actuator 都围绕真实后台系统组织；第三，风险边界有证据，安全隔离、异步幂等、DLQ 治理和 RAG 评估都有测试或文档支撑。

## 10 分钟版本

### 一、为什么做这个项目

我在准备找开发岗时，希望做一个既能体现 Java 后端基础，又能体现 AI 应用能力的项目。所以我没有做简单的聊天 demo，而是选择了“企业知识服务 + 工单协同”这个更贴近真实企业场景的题目。

### 二、系统的核心业务闭环

1. 管理员创建知识库并导入文档
2. 文档异步解析成 chunk，并建立向量索引
3. 用户在工作台提问
4. 系统检索知识并返回答案
5. 如果未命中或低置信度，则转人工工单
6. 工单解决后生成知识草稿
7. 草稿审核发布后回流知识库
8. 运营人员通过看板持续分析知识缺口

### 三、我重点做的技术点

#### 1. 检索质量

- 查询扩展，把中英文技术词、业务别名统一起来
- 混合检索，把 lexical score 和 vector score 做融合
- 对标题命中、章节命中、靠前 chunk 做额外加权
- 控制单文档返回 chunk 数，避免检索结果被一个文档垄断
- 增加低置信度转人工逻辑，避免答非所问

#### 2. 异步任务与治理

- 文档解析和索引独立成异步任务
- 接入 RabbitMQ 做消息驱动
- Redis 做运行时锁和状态跟踪
- Dead Letter Queue 做失败治理
- 支持自动重试和人工回放

#### 3. 工程化

- Flyway 管理数据库版本
- Docker Compose 管理基础设施
- 提供 CI 工作流
- 补了集成测试和检索评估脚本

### 四、如果继续演进

- 接入更强的 embedding 模型
- 引入专业向量数据库
- 增加召回评测集和质量监控
- 补接口限流、熔断、Prometheus 指标
