# Spring Security + JWT + RBAC 知识框架

## 1. 这部分为什么重要

很多初学者会把“登录功能”理解成前端输账号密码，后端查库返回成功。

但企业项目真正关心的是：

- 用户是谁
- 是否已登录
- 有什么权限
- 能不能访问这个接口
- 是否能看到别的租户的数据

这就是鉴权链路。

## 2. 这套项目里的安全目标

当前项目至少要解决：

- 管理员和普通用户身份区分
- 登录后接口访问控制
- 基于角色的能力边界
- 租户间数据隔离
- 非法请求拦截

## 3. 核心知识地图

### 3.1 认证与授权

- 认证：你是谁
- 授权：你能做什么

### 3.2 Session 与 Token

传统方案：

- 服务端保存会话

当前项目方案：

- JWT Token

JWT 的好处：

- 无状态
- 适合前后端分离
- 容易携带用户身份信息

### 3.3 Spring Security 基础

你要理解这些角色：

- Security Filter Chain
- Authentication
- UserDetailsService
- PasswordEncoder
- 自定义过滤器

### 3.4 JWT 关键点

- Header
- Payload
- Signature

你至少要知道：

- JWT 不是加密本身，而是签名校验
- 不要把敏感信息直接明文塞进 token
- token 需要有过期时间

### 3.5 RBAC

RBAC = Role Based Access Control

核心关系：

- 用户
- 角色
- 权限

在这个项目里，你可以先按简化模型理解：

- 平台管理员
- 租户管理员
- 普通用户

## 4. 项目里的真实落点

### 相关模块

- `src/main/java/com/knowflow/auth`
- `src/main/java/com/knowflow/tenant`
- `src/main/java/com/knowflow/common/security`

### 典型链路

1. 用户登录
2. 后端校验账号密码
3. 生成 JWT
4. 前端后续请求带上 `Authorization: Bearer xxx`
5. JWT 过滤器解析 token
6. 把当前用户信息放进上下文
7. 后续业务按角色和租户做限制

### 多租户隔离

这是这个项目里非常重要的点。

即使两个租户都是管理员，也不能互相看到对方的数据。

所以很多查询会天然带上：

- 当前用户所属租户 id

## 5. 你要特别理解的工程点

### 5.1 为什么不能只做前端权限控制

因为前端可以被绕过。

真正有效的权限控制必须在后端。

### 5.2 为什么租户隔离重要

这个项目本质上是一个企业级 SaaS 系统雏形。

SaaS 场景下最重要的安全问题之一就是：

- A 企业的数据不能被 B 企业看到

### 5.3 为什么要统一当前用户上下文

如果每个接口都手动传 userId、tenantId，会很乱。

更好的方式是：

- 登录后从 token 解析出身份
- 放到统一的 `CurrentUser` 上下文里

可以这样说：

“在 KnowFlow 项目里，我把登录、鉴权和租户隔离做成了一条完整链路。登录成功后签发 JWT，请求经过安全过滤器解析 token，再把当前用户信息放到上下文中。后续像知识库、文档、工单、问答这些业务查询都会基于当前租户做隔离，同时配合角色控制不同接口的访问范围。”
