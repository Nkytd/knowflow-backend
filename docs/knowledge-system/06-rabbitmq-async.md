# RabbitMQ + 异步任务知识框架

## 1. RabbitMQ 在项目里解决什么问题

这个项目里最典型的异步场景是：

- 上传文档后解析
- 文档解析后索引
- 索引完成后可检索
- 失败后重试
- 最终进死信队列

如果这些动作都同步执行，会有明显问题：

- 用户上传文档要等很久
- 链路耦合太重
- 一个环节失败会拖垮整个请求

RabbitMQ 的作用就是把“立即响应”和“后台处理”拆开。

## 2. 你要先理解异步

同步：

- 请求发出后，必须等任务做完才能返回

异步：

- 请求先快速返回
- 后台再慢慢处理任务

### 3.4 可靠性设计

## 4. 这个项目里的真实业务链路

### 文档处理链路

1. 用户上传文档
2. 系统创建解析任务
3. 消息进入队列
4. 解析 worker 消费任务
5. 生成知识分片
6. 推进索引任务
7. 索引完成后文档进入可检索状态

### 失败治理链路

1. 任务消费失败
2. 记录失败次数
3. 自动重试
4. 超过阈值进入 DLQ
5. 后台页支持回放

## 6. 你要特别理解的几个概念

### 6.1 幂等性

同一个消息如果被重复消费，结果应该仍然正确。

例如：

- 同一个解析任务不能重复无限创建分片

### 6.2 死信队列

当消息无法正常处理时，不直接丢掉，而是放到专门队列里，方便排查和回放。

### 6.3 重试

失败不代表永久失败。

可能原因只是：

- 下游暂时不可用
- 网络短暂波动
- 资源暂时不足
