# RAG / 检索 / 知识服务知识框架

## 1. 这是这个项目最有亮点的部分

如果只做一个普通后台，它的竞争力主要在：

- Java 基础
- Spring Boot
- 数据库

而你的项目还有一个亮点：

- 知识检索与问答协同

这意味着你不仅能讲传统后端，还能讲：

- 知识库
- 文档解析
- 分片
- 检索
- 问答
- 工单回流

## 2. 先理解 RAG 是什么

RAG = Retrieval-Augmented Generation

你可以先把它理解为：

1. 先检索相关知识
2. 再基于知识生成答案

它和纯大模型裸问答的区别是：

- 纯模型更容易胡说
- RAG 更强调“答案要基于知识库证据”

## 3. 这个项目里的完整知识服务链路

### 3.1 知识进入系统

- 创建知识库
- 上传文档
- 存储到 MinIO
- 创建解析任务

### 3.2 文档处理

- 读取文档
- 归一化文本
- 切片
- 生成 `knowledge_chunk`

### 3.3 检索阶段

- 用户提问
- 召回相关 chunk
- 计算召回分数
- 选出 topK

### 3.4 回答阶段

- 把检索结果作为上下文
- 生成答案
- 返回来源片段

### 3.5 没命中怎么办

- 标记 `NO_HIT`
- 转人工
- 生成工单

### 3.6 问答之后怎么办

- 工单处理
- 形成知识草稿
- 回流知识库

这就是一个闭环系统，而不是单点问答 demo。

## 4. 核心知识地图

### 4.1 文档切片

为什么要切片：

- 文档太长不能直接全喂给模型
- 检索粒度太粗会影响准确率

切片要考虑：

- 长度
- 语义完整性
- 召回效果

### 4.2 召回

召回就是：

- 从知识库里找出可能相关的内容

常见方式：

- 关键词召回
- 向量召回
- 混合召回

### 4.3 排序

召回不等于最终就用。

还要排序，选出最相关的几个片段。

### 4.4 命中与未命中

一个真正可用的知识系统必须考虑：

- 什么时候可以回答
- 什么时候应该诚实地说“我不知道”

### 4.5 来源可追溯

这也是你的项目很重要的一点：

- 返回答案时给出来源片段

这样做的价值：

- 可解释
- 方便人工核验
- 有利于后续优化召回质量

## 9. 你怎么结合项目回答

可以这样说：

“我的项目不是简单接一个大模型接口，而是做了完整的知识服务链路。文档上传后会进入解析和切片流程，系统基于切片结果做检索，再把召回结果作为上下文提供给问答模块。如果问题没有命中，就转为人工工单；工单处理完成后再沉淀成知识草稿回流知识库。这样系统会越用越完整。”
